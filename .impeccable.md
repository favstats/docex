# Design Context

## Target Audience
Developers and academics who work with .docx files programmatically. Primarily social scientists, political communication researchers, and technical writers who need to respond to peer review, format manuscripts for journals, and manage document workflows from the terminal or through AI agents.

## Use Cases
- Decompile .docx to .dex plain text, edit, compile back
- Respond to peer review with tracked changes via CLI/API
- Format manuscripts for specific journal presets
- AI agents editing Word documents through MCP
- Interactive playground for exploring .dex format

## Brand Personality
Technical precision meets academic rigor. Like LaTeX but for Word. The tool should feel powerful yet approachable -- a serious developer tool that academics aren't intimidated by. Confidence without arrogance. Clean, sharp, purposeful.

## Design Direction
**Tone**: Editorial/utilitarian hybrid -- the feeling of a well-typeset academic journal meets a modern developer tool. Think: Stripe docs meet Overleaf.
**Differentiation**: The split-pane .dex editor with live document preview is the hero moment. Showing the raw text format alongside a beautiful rendered document is the "wow" shot.
**Color**: Warm accent on a dark or neutral ground. Avoid the generic blue-on-dark developer aesthetic.
