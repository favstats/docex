# v0.3.0 Design: Stable Addressing + Usability + Robustness

> Spec for the next docex release. Covers the new addressing system (paraId-based),
> usability improvements (colored diff, dry-run, doctor), and robustness (validate,
> backup, lock file, fuzzy retry).

## 1. Stable Addressing System

### Problem

Current docex finds elements by text matching against the full document. This fails when:
- The same text appears in multiple paragraphs
- Previous operations change content, breaking subsequent matches
- Insertions shift everything, making indices unreliable

### Design

Three-tier addressing. Paragraph-level uses OOXML's built-in `w14:paraId` (stable, unique, survives mutations). Sub-paragraph uses text match, run id, or character offset.

#### Tier 1: Paragraph identity via w14:paraId

Every `<w:p>` element has (or gets) a `w14:paraId` attribute, an 8-char hex string unique within the document.

On `doc.map()`:
- Parse all paragraphs, extract their `w14:paraId`
- If any paragraph lacks one, inject a random hex id
- Return structured tree with ids on every element

On `doc.id(paraId)`:
- Find the `<w:p>` with that `w14:paraId` in the XML
- Return a ParagraphHandle object for chained operations

#### Tier 2: Sub-paragraph addressing (within a paragraph found by id)

Three options, all available on a ParagraphHandle:

**A. Character offset** (for programmatic/regex use):
```js
doc.id("3A2F0021").replaceAt(45, 52, "300,000");
```

**B. Run id** (for targeting specific formatted spans):
```js
doc.id("3A2F0021").run("A1B2C3D4").bold();
```
Uses `w14:textId` on `<w:r>` elements.

**C. Text match** (default, for most use cases):
```js
doc.id("3A2F0021").replace("268,635", "300,000");
doc.id("3A2F0021").replace("268,635", "300,000", { nth: 2 });
```

#### Tier 3: Convenience shortcuts (backwards compatible)

The old text-matching API still works for simple cases and CLI:
```js
doc.replace("268,635", "300,000");  // scans whole document, first match
doc.afterHeading("Methods").insert("text");  // explicit heading match
doc.afterText("We found that").insert("text");  // explicit body match
```

#### doc.map() return structure

```js
{
  sections: [
    {
      heading: { id: "3A2F001B", text: "Introduction", level: 1 },
      paragraphs: [
        { id: "3A2F001C", text: "This is the first...", type: "body" },
        { id: "3A2F001D", text: "The second paragraph...", type: "body" },
      ],
      figures: [
        { id: "3A2F0025", number: 1, caption: "Figure 1. Classification funnel" },
      ],
      tables: [],
      comments: [...],
    },
    // ...
  ],
  allParagraphs: [...],
  allFigures: [...],
  allTables: [...],
  allComments: [...],
}
```

#### ParagraphHandle API

```js
const handle = doc.id("3A2F001C");

// Text operations (Tier 2C: text match within this paragraph)
handle.replace(old, new, opts)      // opts: { nth, tracked, author }
handle.delete(text, opts)
handle.bold(text, opts)
handle.italic(text, opts)
handle.highlight(text, color, opts)
handle.comment(text, opts)
handle.footnote(text, opts)

// Character offset (Tier 2A)
handle.replaceAt(start, end, newText, opts)

// Run-level (Tier 2B)
handle.run(textId).bold(opts)
handle.run(textId).replace(newText, opts)

// Structural operations
handle.insertAfter(text, opts)
handle.insertBefore(text, opts)
handle.remove(opts)

// Properties
handle.text                // current text content
handle.id                  // the paraId
handle.type                // "heading" | "body" | "caption" | "abstract"
handle.section             // which section this paragraph belongs to
handle.next                // next ParagraphHandle
handle.prev                // previous ParagraphHandle
```

#### AI agent workflow

```js
const doc = docex("manuscript.docx");
const map = await doc.map();

// Step 1: find what you need from the map
const methods = map.sections.find(s => s.heading.text === "Methods");
const targetPara = methods.paragraphs.find(p => p.text.includes("268,635"));

// Step 2: operate by id (bulletproof)
doc.id(targetPara.id).replace("268,635", "300,000");
doc.id(targetPara.id).comment("Verify this number");

// Step 3: insert returns new id
const newId = doc.id(methods.heading.id).insertAfter("New methodology paragraph.");
doc.id(newId).bold("methodology");

await doc.save();
```

## 2. Usability Improvements

### Colored terminal diff for revisions

`docex revisions manuscript.docx` shows:
- Red text with strikethrough for deletions
- Green text for insertions
- Author and date in gray

Uses ANSI codes (already used in CLI for other output).

**File:** Modify `cli/docex-cli.js` revisions listing.

### --dry-run flag

Every mutating CLI command supports `--dry-run`:
```bash
docex replace manuscript.docx "old" "new" --dry-run
# Output: Would replace 3 occurrences in paragraphs 12, 45, 89
# No file modified
```

Programmatic: `doc.save({ dryRun: true })` returns the result object without writing.

**Files:** Modify `src/workspace.js` (skip write on dryRun), `cli/docex-cli.js` (parse flag).

### docex doctor

```bash
docex doctor manuscript.docx
```

Checks:
- Valid zip archive
- document.xml exists and has w:body
- All relationships in document.xml.rels resolve to existing files
- All media files in word/media/ are referenced by at least one relationship
- No orphaned images (in media/ but not in document)
- No broken images (referenced but not in media/)
- Comment infrastructure consistent (comments.xml count matches ranges in document.xml)
- Paragraph count > 0
- File size reasonable (not 0, not > 100MB)
- Heading hierarchy valid (no H3 without preceding H2)
- All w14:paraId values unique (no duplicates)

Output: green checkmarks for passing, red X for failing, with fix suggestions.

**Files:** Create `src/doctor.js`, add CLI command.

### docex init

```bash
docex init
# Creates .docexrc in current directory:
# {
#   "author": "Fabio Votta",
#   "safeModify": "/path/to/safe-modify.sh",
#   "style": "academic",
#   "backup": true
# }
```

docex reads .docexrc (and ~/.docexrc for global defaults) to set defaults.

**Files:** Modify `src/docex.js` (read .docexrc on open), add CLI command.

### doc.preview()

```js
doc.replace("old", "new");
doc.after("Methods").insert("text");
doc.at("anchor").comment("note");

const preview = doc.preview();
// "3 pending operations:
//   1. replace: 'old' -> 'new' (tracked, by Fabio Votta)
//   2. insert after 'Methods': 'text' (tracked, by Fabio Votta)
//   3. comment at 'anchor': 'note' (by Reviewer 2)"
```

**File:** Modify `src/docex.js` (format pending operations).

### doc.find()

```js
const results = await doc.find("268,635");
// [{ id: "3A2F0021", paragraph: 4, section: "Methods",
//    context: "...collected 268,635 advertisements from..." }]
```

**File:** Add to `src/paragraphs.js` or `src/docmap.js`.

### doc.structure()

```js
const tree = await doc.structure();
// Introduction (H1)
//   3 paragraphs, 0 figures
//   Methods (H1)
//     Data Collection (H2)
//       2 paragraphs, 1 figure
//     Advertiser Classification (H2)
//       3 paragraphs
//   Results (H1)
//     4 paragraphs, 3 figures, 2 tables
//   Discussion (H1)
//     5 paragraphs
```

CLI: `docex structure manuscript.docx`

**File:** Part of `src/docmap.js`.

### doc.explain()

```js
doc.explain("268,635");
// Paragraph 4 (id: 3A2F0021), section "Methods":
//   Run 1: <w:r><w:rPr><w:rFonts.../></w:rPr><w:t>...collected </w:t></w:r>
//   Run 2: <w:r><w:rPr><w:b/></w:rPr><w:t>268,635</w:t></w:r>  <-- HERE
//   Run 3: <w:r><w:t> advertisements...</w:t></w:r>
// The text "268,635" is in run 2 (bold, Times New Roman 12pt)
```

For debugging why an operation might fail (run splitting, entities, etc.)

**File:** Add to `src/docmap.js` or `src/paragraphs.js`.

## 3. Robustness Improvements

### Automatic backup before every save

Every `doc.save()` creates a timestamped backup in a `.docex-backups/` directory next to the file, regardless of whether --safe is used:

```
manuscript.docx
.docex-backups/
  manuscript_2026-03-27_163045.docx
  manuscript_2026-03-27_170112.docx
```

Keeps last 20 backups, auto-prunes older ones.

Can be disabled: `doc.save({ backup: false })` or `"backup": false` in .docexrc.

**File:** Modify `src/workspace.js`.

### doc.validate()

```js
const result = await doc.validate();
// { valid: true, warnings: [], errors: [] }
// or:
// { valid: false, errors: ["Relationship rId15 references missing file image8.png"],
//   warnings: ["Heading hierarchy skips from H1 to H3 at paragraph 45"] }
```

Same checks as `docex doctor` but programmatic. Does not modify the document.

**File:** Part of `src/doctor.js` (shared logic between CLI and API).

### Fuzzy retry on match failure

When a text match fails, before throwing an error:
1. Try case-insensitive match
2. Try with normalized whitespace (collapse multiple spaces)
3. Try with decoded XML entities
4. Show closest matches (already implemented in v0.2.0)

**File:** Modify `src/paragraphs.js` match logic.

### Lock file

When docex opens a file for editing, create `.manuscript.docx.docex-lock`:
```json
{ "pid": 12345, "started": "2026-03-27T16:30:00Z", "user": "fabio" }
```

On open:
- If lock file exists and PID is still running: throw "File is being edited by another docex process"
- If lock file exists but PID is dead: remove stale lock, proceed
- On save/discard: remove lock file

**File:** Modify `src/workspace.js`.

### doc.stats()

```js
const s = await doc.stats();
// { words: { total: 7234, body: 6102, abstract: 285, headings: 67, captions: 240 },
//   paragraphs: 212, headings: 23, figures: 10, tables: 3,
//   citations: 19, comments: 38, revisions: 5,
//   pages: null }  // pages requires PDF rendering, null if unavailable
```

**File:** Combines existing wordCount + list functions.

### doc.contributors()

```js
const c = await doc.contributors();
// [{ name: "Fabio Votta", changes: 12, comments: 3, lastActive: "2026-03-27" },
//  { name: "Mira", changes: 0, comments: 16, lastActive: "2026-03-18" },
//  { name: "Prof. Strict", changes: 0, comments: 4, lastActive: "2026-03-24" }]
```

**File:** New function in `src/revisions.js` or `src/metadata.js`.

### doc.timeline()

```js
const t = await doc.timeline();
// [{ date: "2026-03-18T11:15:00Z", type: "comment", author: "Mira", text: "Strong framing..." },
//  { date: "2026-03-24T10:00:00Z", type: "insertion", author: "Fabio", text: "new paragraph..." },
//  { date: "2026-03-24T10:00:00Z", type: "deletion", author: "Fabio", text: "old text..." }]
```

**File:** Combines comment dates + revision dates, sorts chronologically.

### doc.exportComments("csv")

```js
const csv = await doc.exportComments("csv");
// "id,author,date,text,anchor,resolved\n1,Mira,2026-03-18,Strong framing...,Introduction,false\n..."
```

Also supports "json". Useful for intercoder reliability analysis.

**File:** Add to `src/comments.js`.
